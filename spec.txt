1. Project Overview
This document provides a complete technical specification for expanding the Astro-Consultant MCP (Model Context Protocol) server from its current capability set to a full professional-grade astrology platform suitable for use by practising Jyotishis (Vedic astrologers), Western astrologers, and hybrid practitioners.
The current MCP exposes 19 tools covering natal charts, dashas, yogas, doshas, transits, Ashtakavarga, Shadbala, Panchang, and a few utility functions. While this is a solid foundation, professional astrologers require significantly deeper analytical capability — particularly around divisional (Varga) charts, advanced predictive systems, Jaimini techniques, remedial astrology, and Western prognostication methods.

2.3 Bhava Chalit Chart
The Bhava Chalit chart shows the actual house position of planets when using unequal (Sripati or Placidus-style) house cusps, as opposed to whole-sign houses. Many classical Vedic texts use this for precise house analysis. Planets may shift to an adjacent house in the Chalit chart vs the Rasi chart, which significantly changes interpretation.

Jaimini is a distinct and complete system of Vedic astrology, parallel to the Parashari system. It uses its own Karakas (significators), Chara Dasha, Rasi aspects, and Pada (Arudha) system. 8. Utility & Configuration Enhancements
8.1 Ayanamsa Switcher
Different schools of Vedic astrology use different ayanamsas. The current MCP defaults to Lahiri. Professional astrologers — particularly KP (Krishnamurti Paddhati) practitioners — need to switch ayanamsas.
Ayanamsas to support:
	•	Lahiri (Chitrapaksha) — default, most common in India
	•	Raman — used by B.V. Raman school
	•	Krishnamurti (KP) — mandatory for KP astrologers
	•	Yukteshwar — based on The Holy Science
	•	True Chitra — sideral with fixed star Spica
	•	Fagan/Bradley — Western sidereal
	•	Suryasiddhanta — classical Vedic

10. Implementation Notes & Standards
10.1 Standard Birth Data Object
All tools that accept birth data should accept a consistent parameter schema. Recommend defining a standard BirthData object that can be reused across all tool signatures:
// Standard BirthData Parameters (apply to all tools)
{
  "birth_day":    integer,   // 1–31
  "birth_month":  integer,   // 1–12
  "birth_year":   integer,   // e.g. 1997
  "birth_hour":   integer,   // 0–23 (24hr format)
  "birth_minute": integer,   // 0–59
  "latitude":     float,     // e.g. 27.553
  "longitude":    float,     // e.g. 76.634
  "timezone":     string,    // IANA format: "Asia/Kolkata"
  "name":         string,    // Person's name or identifier
  "ayanamsa":     string?    // Optional: LAHIRI (default), KP, RAMAN, etc.
}

10.2 Consistent Response Envelope
All tool responses should include a standard metadata envelope for debugging and auditability:
{
  "tool": "get_varga_chart",
  "version": "2.0",
  "ayanamsa_used": "LAHIRI",
  "ayanamsa_degrees": 23.8189,
  "computation_time_ms": 42,
  "data": { ... actual response ... }
}

10.3 Error Handling
All tools must return structured errors — never raw HTTP errors or HTML pages (the current implementation returns HTML 404 pages on failure, which is not parseable by AI clients):
// Error response format
{
  "error": true,
  "error_code": "INVALID_BIRTH_DATA",
  "error_message": "birth_hour must be between 0 and 23",
  "tool": "get_varga_chart"
}
